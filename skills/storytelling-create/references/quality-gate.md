# Deck Readiness Reference

Use this reference near the end of creation work, after the deck skeleton, takeaway titles, word design, data flow, image choices, navigation, and appendix map have been drafted.

## Core principle

Do not polish a deck whose structure is not defensible.

The practical implication:

- test the story before production,
- test audience attention before adding more content,
- test visual purpose before polishing visuals,
- and test appendix support before relying on it in discussion.

## 1. Story integrity

Ask:

- Does every main-story slide support the Big Idea?
- Does the sequence move from context to tension to evidence to implication or action?
- Are there slides that exist only because work was done?
- Are there repeated points that should be merged?
- Is the desired action or decision visible by the end?

Fix structure before slide design if the answer is weak.

## 2. Audience action

Ask:

- Who needs to do what after this presentation?
- What decision, discussion, belief shift, or action is the deck designed to create?
- Which slides directly move the audience toward that outcome?
- Which slides only inform without changing the path?

If the desired action is vague, return to planning before production.

## 3. Horizontal logic

Read only the slide titles.

Check whether the titles show:

- context,
- tension,
- evidence,
- implication,
- recommendation,
- and next action.

The title-only story should be coherent enough that someone can understand the argument without seeing the body content.

## 4. Attention contracts

For each slide, decide whether the audience should:

- read,
- scan,
- listen,
- discuss,
- compare,
- inspect,
- decide,
- or remember.

Failure pattern:

- the presenter talks while the audience must read,
- the audience inspects while the presenter moves ahead,
- or the slide contains speaker memory instead of audience support.

## 5. Evidence burden

Ask:

- What proof does the audience need in the main story?
- What proof can live in the appendix?
- Which skeptical questions are likely?
- Which assumptions, caveats, methods, or sensitivities need backup?
- Is the deck over-proving obvious points and under-proving contested ones?

Evidence should be sufficient for trust without turning the main path into an analysis dump.

## 6. Visual purpose

For each graph, image, diagram, or table, ask:

- What job does this visual do?
- Where should the audience look first?
- What should they understand after seeing it?
- What can be removed, muted, or moved to notes?
- Does the title tell the audience what to see?

If the visual has no specific job, remove it or move it to appendix.

## 7. Delivery fit

Check whether the deck is for:

- live presentation,
- virtual presentation,
- executive discussion,
- workshop,
- sales or stakeholder persuasion,
- or standalone reading.

Adjust accordingly:

- live decks can rely more on speaker delivery,
- virtual decks need stronger pacing and orientation,
- standalone documents need more complete labels, captions, and context,
- discussion decks need clear prompts and navigable backup.

## 8. Appendix readiness

Ask:

- Is the appendix organized around likely questions?
- Can the presenter find backup material quickly?
- Are methods, caveats, assumptions, and raw detail separated cleanly?
- Are appendix titles clear enough to navigate during discussion?
- Is anything in the appendix actually essential to the main story?

The appendix is a credibility system, not a dumping ground.

## Final readiness checklist

- [ ] The Big Idea is visible in the structure.
- [ ] The desired audience action is clear.
- [ ] The title-only story works.
- [ ] Each slide has an attention contract.
- [ ] Main-story evidence is sufficient but not overloaded.
- [ ] Each visual has a specific job.
- [ ] Delivery mode has been considered.
- [ ] Appendix support is navigable.
- [ ] Production work can begin without hiding structural uncertainty.
