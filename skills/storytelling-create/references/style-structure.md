# Style & Structure Reference

This reference distills Chapter 5 of *Storytelling with You* into a reusable creation workflow.

## Core principle

Do not let presentation software pull the work directly into production.

The practical implication:

- set style before designing individual slides,
- build the deck skeleton before adding content,
- write titles that state points,
- test the title-only story,
- and move credibility detail into the appendix when it would slow the main path.

## 1. Slides support the presenter

Live presentation slides should support delivery rather than become the document.

Use slides to:

- guide attention,
- make important ideas visible,
- reinforce the spoken story,
- and reduce the audience's listening burden.

If material must stand alone, make a leave-behind document, appendix, or handout. Do not overload live slides to solve a documentation problem.

## 2. Determine the style

Style includes:

- color,
- font,
- logo,
- layout,
- and overall visual tone.

Choose style based on the audience, topic, and desired response.

### Brand or template

If a simple usable brand template exists, use it.

If the template is visually busy:

- add fewer elements than usual,
- increase contrast for the true point,
- avoid adding decoration on top of decoration,
- and consider small adjustments if allowed.

### Custom style

If no useful template exists:

- start simple,
- avoid default slideware templates when they are noisy,
- choose color intentionally,
- prioritize legible fonts,
- include black, white, and grays for hierarchy,
- and keep the visual system flexible enough for the story.

## 3. Make or adapt the slide master

Use a slide master or template to make the visual system reusable.

Useful layouts:

- title slide,
- content slide with a strong title area,
- divider slides,
- plain white slide,
- plain black slide,
- and any recurring layout required by the story.

Do not overdesign. The slide master should reduce friction and drift, not lock the deck into awkward formats.

## 4. Build the deck skeleton

A deck skeleton is the low-fidelity version of the presentation inside the slide tool.

It may include:

- slide titles,
- rough sections,
- divider or navigation slides,
- placeholder slides,
- speaker notes for details to preserve,
- and appendix sections.

The skeleton keeps structural feedback alive after planning moves into slideware.

## 5. Use takeaway titles

A descriptive title names the topic.

Examples:

- "Customer preferences"
- "Ingredient cost"
- "Packaging test"

A takeaway title states the point.

Examples:

- "Customers prefer the original mix"
- "Macadamia costs threaten margin"
- "Packaging changes alone will not solve the problem"

Working rule:

- identify the main point of the slide,
- state it as a sentence,
- make it concise,
- and use it as the slide title.

## 6. Test horizontal logic

Horizontal logic is the test of reading only the titles to see whether the deck has a story.

Ask:

- Do the titles connect?
- Does the sequence move?
- Is the tension visible?
- Is there enough evidence?
- Does the ending point to action?
- Are there repeated or disconnected slides?

If the title-only version is weak, the problem is structure, not polish.

## 7. Use navigation when needed

A navigation scheme is the deck's map.

Use it when:

- the presentation is long,
- the story has several sections,
- the audience needs orientation,
- or transitions may otherwise feel abrupt.

Good navigation:

- appears near the beginning,
- returns during transitions,
- is visually distinct from body slides,
- and helps the audience understand where they are.

Skip it when it adds ceremony without improving orientation.

## 8. Use appendix detail for credibility

Appendix detail can preserve credibility without overloading the main presentation.

Move detail to the appendix when it:

- proves the work was robust,
- answers likely challenges,
- contains method or backup analysis,
- matters only to some stakeholders,
- or would slow the main story.

The appendix is a credibility tool, not a junk drawer. Organize it so the presenter can navigate it during questions.

## Creation checklist

- [ ] The live deck is not trying to become the whole document.
- [ ] Style supports the topic, audience, and tone.
- [ ] The slide master or template reduces visual drift.
- [ ] The deck skeleton exists before detailed content.
- [ ] Slide titles are takeaway titles.
- [ ] The title-only story passes horizontal logic.
- [ ] Navigation is included only if it helps orientation.
- [ ] Appendix material is organized around likely credibility needs.
